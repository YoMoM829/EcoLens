# Infrastructure overview

## Purpose

This document describes the final cloud infrastructure for Aussie EcoLens, the component responsibilities, and the logical data flow. It also outlines security and deployment considerations aligned with the assignment requirements.

## Final architecture decision

- **AWS** hosts the public-facing edge:
  - Static frontend hosting (S3 frontend bucket + CloudFront).
  - Authentication (Cognito User Pool).
  - Object storage (six S3 buckets — see below).
  - Serverless compute (Lambda — API Lambda as ZIP, ML Lambda as container image via ECR) and RESTful API (API Gateway HTTP API v2).
  - Tag-based notifications (SNS).
  - Secrets management (Secrets Manager — OCI API signing key).

- **Oracle Cloud (OCI)** hosts the metadata store only:
  - Oracle NoSQL Database for media metadata, checksums, tags, counts, and tag-subscription records.

- **Terraform** (`infra/terraform/`) provisions all AWS and OCI resources repeatably, with CI/CD pipelines applying changes.

## Core components and responsibilities

- **Frontend (S3 + CloudFront)**  
  Hosts the React/Vite SPA, which handles sign-in/sign-up, upload, search, tag edits, and media browsing.

- **Authentication (Cognito)**  
  Manages user registration, email verification, sign-in, and JWT issuance. All protected APIs validate Cognito JWTs.

- **Object storage (six S3 buckets)**

  | Bucket | Purpose | Versioning |
  |---|---|---|
  | **uploads** | Original user media. Browsers upload via presigned PUT URLs. `ObjectCreated` events trigger the ML Lambda. | Yes |
  | **thumbnails** | Image thumbnails generated by the ML Lambda. | No |
  | **detections** | Raw MegaDetector detection JSON per processed file. | No |
  | **models** | MegaDetector and species-classifier weight files. ML Lambda downloads on cold start via `MODEL_S3_KEY` env var. | Yes |
  | **frontend** | Compiled React/Vite SPA. Served only through CloudFront; never public. | Yes |
  | **query-temp** | Ephemeral scratch space for query-by-file reference uploads. Files are deleted immediately after ML processing and never written to the database. | No |

- **API (API Gateway HTTP API v2 + Lambda)**  
  Exposes a RESTful API via a `$default` catch-all route. The gateway JWT authorizer validates Cognito tokens before Lambda is invoked. Mangum/FastAPI handles all routing:
  - `GET /health` — public health probe
  - `GET /users/me` — authenticated user info
  - `POST /uploads` — presigned S3 PUT URL + two-layer dedup check (DB + S3 `head_object`)
  - `GET /media?tag=koala:2&tag=wombat:1` — search by tag counts (AND, min count)
  - `GET /media?species=koala&species=dingo` — search by species (AND, count ≥ 1)
  - `GET /media/{file_id}` — resolve file_id → full-size URL
  - `POST /media/similar/presign` — get presigned URL for reference file upload
  - `POST /media/similar` — find similar by pre-uploaded reference file
  - `POST /media/tags` — bulk add / remove tags
  - `DELETE /media` — delete files, thumbnails, and DB records (ownership enforced)
  - `GET /subscriptions` — get current SNS subscription (species + status)
  - `POST /subscriptions` — subscribe / update species-watch email notifications
  - `DELETE /subscriptions` — cancel current SNS subscription

- **Serverless compute (Lambda — two separate deployments)**
  - **API Lambda** (`ecolens-api`): deployed as a **ZIP package**. Entry-point `backend.src.main.handler` (Mangum ASGI adapter). Handles all HTTP endpoints; invokes the ML Lambda for video similarity queries via `lambda:InvokeFunction`.
  - **ML processor Lambda** (`ecolens-s3-handler`): deployed as a **container image** via ECR (built from `ml-service/Dockerfile`, which bundles PyTorch, MegaDetector, and `ml_pipeline`). Triggered by S3 `ObjectCreated` events — reads uploader identity from S3 object metadata (`x-amz-meta-user-id`), checks for duplicates (two layers: OCI NoSQL + S3 `head_object`), generates thumbnails, runs ML inference, writes metadata (including `source_url` and `thumbnail_url`) to Oracle NoSQL, publishes SNS notifications.

- **Notifications (SNS)**  
  Publishes tag-based notifications to subscribed users when new media with watched tags is inserted.

- **Metadata store (OCI Oracle NoSQL)**  
  Stores media records: `media_id` (SHA-256 checksum), `user_id`, `file_type`, `status`, `source_url`, `thumbnail_url`, `original_key`, `thumbnail_key`, `detections_key`, `tags`, `animal_detected`, `top_confidence`, `created_at`.

## Logical data flow

1. User registers and signs in via Cognito and obtains JWT tokens.
2. Frontend calls a protected API Gateway endpoint to request a presigned S3 upload URL, passing the JWT.
3. API Lambda validates the JWT, generates a presigned URL for the media bucket, and returns it to the frontend.
4. Frontend uploads the file directly to S3 using the presigned URL.
5. S3 emits an `ObjectCreated` event and invokes the media processor Lambda.
6. The media processor Lambda:
   - Computes a checksum and checks for duplicates.
   - If duplicate, stops processing and updates metadata as needed.
   - If new:
     - Generates a thumbnail for images and writes it to S3.
     - Extracts 1 fps frames for videos and passes them to the ML worker.
     - Invokes the ML inference worker to detect species and counts.
     - Writes the metadata to Oracle NoSQL.
     - Publishes SNS notifications for matching subscriptions.

7. Frontend calls the query APIs (via API Gateway + API Lambda) to search Oracle NoSQL by tags, species, thumbnail URL, or by uploading a reference file.
   - For query-by-file: the API Lambda places the reference file in the **query-temp** bucket, invokes the ML Lambda (synchronously for images, asynchronously with a job ID for videos), and deletes the temp file immediately after processing. The reference file is never stored permanently or inserted into the database.

## Model versioning

- Pre-trained model artefacts are stored in S3 (or a container registry) with versioned keys.
- The ML pipeline reads the model key and version from environment variables (for example, `MODEL_S3_KEY` and `MODEL_VERSION`).
- Updating the model involves uploading a new `.pt` file and changing environment configuration, without modifying Lambda code.

## Security and best practices

- **Authentication**: All non-signup routes require valid Cognito JWTs.
- **Least privilege (IAM)**:  
  - Lambda roles have only the S3, SNS (6 actions: Publish, Subscribe, Unsubscribe, SetSubscriptionAttributes, GetSubscriptionAttributes, ListSubscriptionsByTopic), CloudWatch, and Secrets Manager permissions they need.
  - NoSQL access is granted only to a dedicated OCI identity used by the backend.

- **Secrets management**:  
  - OCI API private keys and sensitive configuration are stored in AWS Secrets Manager or encrypted Lambda environment variables.
  - No secrets are committed to source control.

- **S3 bucket versioning**:  
  Versioning is enabled on three buckets: uploads, frontend, and models. Thumbnails, detections, and query-temp are excluded — they are auto-generated or ephemeral and can be recreated without versioning overhead.
  - **Uploads**: protects original user media from accidental deletion or overwrite.
  - **Frontend**: allows instant rollback if a bad build is deployed.
  - **Models**: preserves every model artefact so the active model can be rolled back by updating an environment variable, without any code change (satisfies requirement 4.1.1).

- **Network and data security**:  
  - S3 buckets use default server-side encryption.
  - All database traffic uses TLS.
  - Where possible, the path between AWS and OCI is restricted by network rules.

- **Monitoring**:  
  - Lambda logs are sent to CloudWatch.
  - CloudWatch metrics and alarms can be configured for failures and performance issues.

## Deployment notes

Two supported deployment paths:

1. **Console + CLI only (simple)**  
   - Create S3 buckets (uploads, thumbnails, detections, models, frontend, query-temp).
   - Create Cognito User Pool and App Client.
   - Create IAM roles and Lambdas.
   - Wire S3 events and API Gateway routes.
   - Create Oracle NoSQL table and OCI IAM user.
   - Configure Lambda environment variables for OCI and model versioning.

2. **Terraform**  
   - Use `infra/terraform/` to define:
     - AWS S3 buckets, CloudFront, Cognito, ECR repository, Lambda functions (API Lambda as ZIP, ML Lambda as container image), API Gateway HTTP API v2, SNS topic.
     - OCI Oracle NoSQL table and policies (using the OCI provider).
   - Terraform builds and pushes the Docker image to ECR automatically during `terraform apply`.
   - Store Terraform state securely (e.g. S3 backend with state locking via OCI Object Storage or a remote backend of your choice — note: DynamoDB locking is AWS-only and not used here since the application database is OCI NoSQL).
   - Drive deployment via GitLab CI/CD.
