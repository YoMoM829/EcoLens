# Aussie EcoLens

Aussie EcoLens is a multi-cloud, serverless wildlife media platform for uploading, automatically tagging, and querying wildlife images and videos.

The system is designed for citizen science and research: users upload media, a pre-trained model detects species, and the platform stores searchable metadata across AWS and Oracle Cloud.

### Access model

EcoLens operates as a **shared platform**: all uploaded files are visible to every registered user. Any authenticated user can search (by tag counts, species, thumbnail URL, or by uploading a reference file), view full-size images, and add or remove tags on any file. **Deletion is restricted to the original uploader** — only the user who uploaded a file may delete it. Species-watch notifications are also shared: a subscriber is emailed whenever a matching file is uploaded by any user.

## Repository layout

The project is organised into four main areas:

- `frontend/` – React/Vite single-page app for sign-in, upload, search, tag editing, and media browsing.
- `backend/` – FastAPI-based backend running on AWS Lambda behind API Gateway for all REST APIs.
- `ml-service/` – Species detection and classification library (`ml_pipeline`) used by the backend S3 tagging handler.
- `infra/` – Architecture notes, multi-cloud deployment guidance, and Terraform configuration for repeatable provisioning.

Additional folders:

- `scripts/` – Helper scripts for packaging, deploying, and smoke-testing.
- `docs/` – Team and individual reports for the assignment.

## Multi-cloud architecture

The platform spans **AWS** and **Oracle Cloud Infrastructure (OCI)** following a serverless, event-driven design.

### AWS

| Service | Role |
|---|---|
| **Cognito** | User registration, email verification, login, and sign-out. |
| **API Gateway** (HTTP API v2) | REST API edge — all browser API calls are routed here and forwarded to the API Lambda. |
| **Lambda — API** (ZIP) | FastAPI app via Mangum. Handles all REST endpoints, reads/writes OCI NoSQL, and invokes the ML Lambda synchronously (images) or asynchronously (videos) for query-by-file requests. |
| **Lambda — ML** (container via ECR) | Triggered by S3 `ObjectCreated` events on the uploads bucket. Runs MegaDetector inference, generates image thumbnails or extracts one frame per second for videos, writes metadata and tags to OCI NoSQL, and publishes SNS notifications. Deployed as a container image to accommodate PyTorch's size. |
| **ECR** | Stores the ML Lambda container image. |
| **S3 — uploads** | Stores all original media. Browsers upload directly via presigned URLs (bypassing API Gateway's payload limit); `ObjectCreated` events trigger the ML Lambda automatically. |
| **S3 — thumbnails** | Stores image thumbnails generated by the ML Lambda. |
| **S3 — detections** | Stores raw MegaDetector detection JSON for each processed file. |
| **S3 — query-temp** | Ephemeral scratch space for query-by-file uploads. Files are deleted immediately after ML processing and are never persisted to the database. |
| **S3 — models** | Stores MegaDetector model weights. The ML Lambda downloads the model at cold start via an environment variable, enabling model upgrades without code or container changes. |
| **S3 — frontend** | Hosts the compiled React/Vite SPA. Served exclusively through CloudFront; never publicly exposed directly. |
| **CloudFront** | CDN edge for the static frontend. |
| **SNS** | Tag-based email notifications. The ML Lambda publishes when new species are detected; the API Lambda manages subscriptions on behalf of authenticated users. |
| **Secrets Manager** | Stores the OCI API signing private key. Retrieved once at deploy time and injected into both Lambdas as an environment variable. |

### Oracle Cloud Infrastructure (OCI)

| Service | Role |
|---|---|
| **Oracle NoSQL Database Cloud Service** | Sole metadata store. Holds all media records (media_id, owner, S3 keys, file type, tags with counts, status, timestamps). Accessed by both Lambda functions. |

### AWS–OCI secure connection

The OCI private key (RSA PEM) is stored in AWS Secrets Manager, retrieved once at deploy time, and injected as an environment variable. At runtime, both Lambdas construct an OCI API key signer from this in-memory value — the key is never written to disk or baked into any image or ZIP. All OCI NoSQL API calls are signed per the OCI HTTP Signature v1 protocol over HTTPS, providing mutual authentication and transport encryption between AWS Lambda and OCI.

### Upload and processing flow

1. Browser requests a presigned S3 PUT URL from the API Lambda.
2. Browser computes a SHA-256 checksum and checks for duplicates before uploading.
3. Browser uploads file bytes directly to the S3 uploads bucket (bypassing API Gateway).
4. S3 fires an `ObjectCreated` event to the ML Lambda.
5. ML Lambda runs MegaDetector inference, generates thumbnails (images) or extracts frames (videos), writes metadata to OCI NoSQL, and publishes SNS notifications to matching subscribers.

## Local development

### Prerequisites

- Python 3.10+
- Node.js (LTS) and npm
- AWS CLI configured (for deployment and S3 access)
- OCI CLI or access to the OCI Console (for NoSQL table and credentials)

### Backend API (FastAPI)

```bash
cd backend
python -m pip install -r requirements.txt

# Run the FastAPI app locally on http://localhost:8000
uvicorn src.main:app --reload
```

### ML pipeline (inference library)

```bash
cd ml-service
python -m pip install -r requirements.txt

# Save a backend-compatible metadata JSON record locally
python tests/test_local.py path/to/file.jpg --user-id testuser

# Test the full S3 upload → tag → metadata flow (backend handler)
cd ..
python backend/tests/test_local_s3_trigger.py
```

`process_file()` in `ml-service` returns the same metadata shape that `backend/src/repository.put_file_record()` persists (file_id, user_id, file_type, S3 keys, tags map, status, etc.). See `ml-service/README.md` for the full schema.

### Frontend (React/Vite)

```bash
cd frontend
npm install
VITE_API_BASE_URL=http://localhost:8000 npm run dev
```

Then open the URL printed by Vite (default `http://localhost:5173`).

## Deployment overview

A typical production deployment uses:

- **Two Lambda functions** deployed differently:
  - **API Lambda** (`ecolens-api`) — deployed as a **ZIP package**. Contains only FastAPI/Mangum and lightweight dependencies; no ML model is bundled.
  - **ML Lambda** (`ecolens-s3-handler`) — deployed as a **container image** via Amazon ECR. PyTorch and MegaDetector exceed the 250 MB zip limit, so a container is required. Built from `ml-service/Dockerfile`, which bundles PyTorch, MegaDetector, `ml_pipeline`, and all inference dependencies.
- **Oracle NoSQL** created in an OCI compartment, accessed by backend Lambdas via OCI API signing keys.
- **Frontend build** uploaded to an S3 bucket and served via CloudFront.

See `infra/README.md` and `infra/architecture.md` for the detailed architecture diagram and step-by-step deployment runbook.
